## A. payer (결제) — mock이 가린 실전 갭

우선순위 = 급한 불 순서.

### 1순위 — 정확성 (지금도 살짝 버그)
- [ ] **실제 금액 상한 체크**: 지금 `guard`는 카탈로그 추정가로만 검사. real 모드는 402가 부르는 실제 금액을 냄. → 402 requirements의 실제 금액을 서명 *전에* 파싱해 상한과 비교.
- [ ] **결제 전 잔액 확인**: 지갑 USDC 잔액 조회 후 부족하면 미리 중단(라이브에서 겪은 실패 예방).

### 2순위 — 신뢰성
- [ ] **tx 확정 대기**: 200 받아도 온체인 confirmation 안 기다림. tx hash 폴링으로 진짜 정산 확인.
- [ ] **에러 처리**: 잔액부족·facilitator 다운·v1/v2 봉투 불일치 케이스별 처리(재시도·폴백·안내).

### 3순위 — 운영
- [ ] **회계/영수증 저장**: receipt 출력만 하고 버림 → 세션·모델별 지출 원장 + 정산.
- [ ] **키 관리**: `.env` 평문(일회용 테스트넷이라 OK) → 실자금은 KMS/시크릿볼트/HSM.
- [ ] **동시성/지갑 풀**: 단일 지갑·순차 가정. 병렬 결제 시 nonce 충돌 방지, 지갑 풀·자동 충전.

## B. selector (선택) — 자연어 우선순위

- [ ] **NL 프롬프트 → 우선순위 자동 추론**: 지금은 `--priority coding` 명시 필요. 자연어("러스트 코드 짜줘")를 LLM이 분석 → coding 우선순위 자동 생성. `--priority` 없이 작동.
- [ ] 현재: prompt(작업 텍스트)는 selector가 안 읽음. 모델 고른 뒤 그 모델한테 던질 뿐. NL 추론 붙이면 prompt도 선택에 반영.

## C. catalog (마켓플레이스) — 정적 → 동적

3레벨. 아래로 갈수록 큼.

- [ ] **레벨 1 (쉬움, ~반나절)**: 동적 카탈로그. `catalog.yaml` 하드코딩 → Artificial Analysis API에서 모델 목록 실시간 수집(selector `--live` 재사용). 프록시에 `/marketplace` 목록 엔드포인트 추가.
- [ ] **레벨 2 (중간, ~2~3일)**: 다중 셀러 시뮬. 프록시 여러 개(같은 모델 다른 가격·평판) + discovery 레지스트리. 구매자가 셀러 비교·선택 → 진짜 마켓 경쟁.
- [ ] **레벨 3 (큼, ~1주+)**: 온체인 리스팅. ERC-8004 Identity Registry를 마켓 등록부로. 셀러 온체인 등록 + 온체인 평판까지 봐서 선택. 완전 탈중앙.

권장: 레벨1 → 레벨2 순.
